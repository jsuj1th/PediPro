# FormFiller Repository-Wide Change Proposal (PDF-First AcroForm Flow)

Status: Proposed (awaiting approval)  
Version: 0.1  
Date: 2026-03-17  
Owner: Product + Engineering

## 1. Requested Target Flow

This proposal implements the exact flow you specified:

1. Staff uploads a base PDF.
2. Staff uses an in-product editor to create AcroForm fields (`field_id`, `field_name`, `type`, etc.).
3. Published template becomes active for patient intake.
4. Patient is asked questions using `field_name` labels from that template.
5. Patient responses are stored by `field_id` (stable key) with metadata.
6. Staff exports:
- canonical JSON response payload
- filled PDF generated by AcroForm field-name mapping (`patientname_patientregistration.pdf`)

## 2. Product-Level Shift

Current model in repo is mostly JSON-template-driven forms with manual PDF overlay support.  
Proposed model becomes **PDF-first and field-driven**:

- source of truth for intake form structure = published AcroForm template + field metadata
- patient UI generated from published field metadata
- persistence keyed by `field_id`
- PDF export uses AcroForm fill (field names), not coordinate overlays

## 3. Canonical Data Contracts

## 3.1 Field Definition

Each template field record:

- `field_id` (UUID/string, immutable key)
- `field_name` (human label shown to patients/staff)
- `field_type` (`text`, `textarea`, `checkbox`, `radio`, `select`, `date`, `signature`)
- `acro_field_name` (actual AcroForm field name in PDF)
- `required` (boolean)
- `page_number` (int)
- `x`, `y`, `width`, `height` (editor coordinates)
- `options_json` (for radio/select/checkbox groups)
- `validation_json` (length, regex, allowed values)
- `display_order` (for patient form generation)
- `section_key` (grouping in patient/staff UI)

## 3.2 Patient Response Storage

Submission responses stored as:

```json
{
  "submission_id": "uuid",
  "template_id": "uuid",
  "template_version": 3,
  "responses": {
    "field_001": { "value": "Jane", "updated_at": "..." },
    "field_002": { "value": "Doe", "updated_at": "..." },
    "field_010": { "value": true, "updated_at": "..." }
  }
}
```

Key rule: response keys are always `field_id`, never label text.

## 5. New End-to-End Flow (Operational)

## Step A: Template Authoring (Staff Admin)

1. Upload base PDF.
2. Open editor and place fields.
3. Configure each field (`field_id`, `field_name`, `type`, `required`, `acro_field_name`, validations).
4. Save draft.
5. Run preview fill.
6. Publish template version.

## Step B: Patient Intake

1. Parent starts intake session.
2. System fetches active template for practice/visit type.
3. UI asks questions using `field_name`.
4. Autosave persists `{ field_id: value }`.
5. Submit locks completed snapshot with `template_version`.

## Step C: Staff Review/Edit

1. Staff opens submission/patient.
2. Editor renders fields from same template metadata.
3. Staff can update values by `field_id` with audit logging.

## Step D: Export

1. Export JSON returns canonical response map by `field_id`.
2. Export PDF fills AcroForm by `acro_field_name` from stored responses.
3. File downloads as `patientname_patientregistration.pdf`.

## 6. Migration Plan From Current State

## Phase 1: Introduce Template Tables + APIs

- keep current intake working
- add template infrastructure behind feature flag

## Phase 2: Dual-Run Intake

- new sessions can use field-driven templates
- old sessions continue legacy path

## Phase 3: Switch Export to AcroForm Fill

- use published template for PDF export
- keep coordinate overlay fallback for untemplated legacy submissions

## Phase 4: Full Cutover

- default all practices to template-driven intake
- retire legacy hardcoded form rendering progressively

## 7. Acceptance Criteria

- Staff can create/publish a template in-product from uploaded PDF.
- Patient questions are rendered from published fields (`field_name`).
- Response persistence is keyed by `field_id`.
- Staff can edit responses in user-friendly UI.
- PDF export reliably fills AcroForm fields and downloads with required filename format.
- JSON export includes `template_id`, `template_version`, and `field_id` keyed values.

## 8. Risks and Controls

Risk: template misconfiguration creates broken patient forms.  
Control: publish-time validation + preview + required field checks.

Risk: field renaming breaks old data mapping.  
Control: immutable `field_id`; allow `field_name` updates without key changes.

Risk: template version drift between intake and export.  
Control: submission stores `template_version`; export always uses that version.
